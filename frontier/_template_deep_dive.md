## 核心实现拆解（剧透式）

> **何时写**：原文或配套 repo / API / SDK 存在可落地的工程实现时，**必填**本节；纯战略/定价/合作类可写「无工程实现，跳过」。
>
> **写法**：像「剧透」一样点破系统**实际怎么跑**，不要求面面俱到。只写 **2–4 个最核心模块**（例如：加载链路、一次请求的 data flow、状态存在哪、扩展点如何挂接）。避免 API 全表、目录树堆砌。
>
> **建议结构**（按需取用）：
> - **核心模块表**：模块名 | 职责 | 关键产物/接口（各一行）
> - **主路径**：一次典型调用/会话从输入到输出的步骤（可用编号或 mermaid sequence）
> - **与「表面叙事」的差异**：官方话术 vs 代码/文档里真实机制（若有）
>
> 体量极大、需贴代码片段时，可另起 `YYYY-MM-DD_<slug>_实现拆解.md`，本节保留摘要并链过去。
