      信源： Building agents with Skills（Anthropic, 2026-01-22） · The evolution of agentic surfaces · Claude for Financial Services · Claude for Life Sciences → Science · Model Context Protocol

      核心论点：Anthropic 从来没在做 N 个平行产品——他们在做 一个统一的 Agent 平台，然后按行业 / 职业打包成不同品牌。所谓 Claude Code / Cowork / Design / Finance / Science / Legal / Gov，全部跑在同一栈上：Claude Model → Agent Loop → Runtime → MCP → Skills。Managed Agents 是这个平台被公开的时刻——它把 Anthropic 内部造 Claude X 的那套底座变成了对外 API。「有没有可能抽象成一个产品」这个问题，他们内部已经答完了：技术上做减法，品牌上做加法。
      一页要点
      • 「Claude X」不是 N 个产品，是 N 张皮。产品差异只集中在 5 条轴：入口 surface / skill pack / MCP 连接器组合 / 多 agent 拓扑 / 合规边界（trust boundary）。
      • 官方自己讲穿了：Anthropic 2026-01 博客明说「we stopped building specialized agents and started building skills instead」。因为「code is all you need」，通用 agent + skill 包 = 专家 agent。
      • Managed Agents = 底座商品化。2023 API → 2025 Agent SDK → 2026-06 Managed Agents。Anthropic 把「造 Claude X 的能力」打包成 API 卖给客户，让摩根/花旗自己也能造 Claude for X。
      • MCP 是未来 2-3 年最关键的 ToB 接口——任何内部系统「能否被 Agent 调用」的通行证，等同当年「有没有 REST API」。
      • 受监管市场需要品牌化容器：金融/医疗/法律/国安不会为「通用 AI 加 plugin」立项，但会为「Claude for X + 合规认证 + partner 生态」立项。所以品牌化是商业必要，不是技术必要。
      • 下一批独立品牌可预测：Sales / Marketing / Health / Accounting / HR — 已经作为 Cowork plugin 存在，只差一个品牌就能变成「Claude for X」。

      一背景：从 Claude Code 到「Claude X 时代」#

      — WHY NOW · 一年时间铺了 8 个品牌

      2025 Q1 Claude Code 破圈之后，Anthropic 的产品发布节奏突然从「季度」压到「月」。到 2026 年年中，「Claude for X」的品牌已经铺到 8 个——Code · Gov · Finance · Life Sciences（后升级为 Science）· Cowork · Design · Legal · Managed Agents。外界的第一反应是「Anthropic 在造 N 个平行产品」；但只要把发布顺序 + 官方架构 + 底层 API 对齐一次，就会看到另一个几何形状：所有产品都在同一根竖轴上生长。

      时间轴上有两条并行的线：

      • 上半轴（灰色 + 紫色实心）= 垂直品牌线：Code → Finance → Life Sciences → Design → Legal → Science。每一个都对应一个受监管 / 高价值行业。
      • 下半轴（紫色实心）= 平台底座线：Cowork（面向所有知识工作者的通用桌面 Agent）→ Managed Agents（把底座打包成 API 给客户）。

      两条线不是独立发育——底座线的每一次跃迁，都直接决定品牌线能不能继续铺。Cowork 让通用 Agent 在桌面跑通、Managed Agents 让「造 Agent 的能力」变成商品——这才有了 2026 下半年 Design / Legal / Science 快速上马的空间。

      三核心论点：一个平台，五张皮#

      — From N specialists to one generalist × N skill packs

      把上面的时间轴和产品清单叠在一起，会得到一个反直觉的结论：不同 Claude X 的技术核心几乎一模一样。真正让「Claude for Finance」和「Claude for Science」看起来不同的，只是它们挂了不同的 skill 包、连了不同的 MCP、开了不同的 surface。

      更妙的是 Anthropic 自己在官方博客里把这件事讲穿了：

      "We used to think agents in different domains would look very different. A coding agent, a research agent, one for finance, one for marketing—each seemed to need its own tools and scaffolding.

      But as models improved... we converged on a different approach. ... We stopped building specialized agents and started building skills instead."

      — Anthropic Blog, 2026-01-22

      四技术底座：Anthropic 官方的 4 层架构#

      — Loop reasons · Runtime executes · MCP connects · Skills guide

      2026-01 的 Skills 博客里，Anthropic 画了这张官方架构图——迄今为止「Claude X 全家桶」的最权威解剖图：

      四层是 Anthropic 官方博客的原文表述，从下往上：Agent loop（决策系统）→ Agent runtime（执行环境）→ MCP servers（外部连接）→ Skills library（领域知识）。所有「Claude X」都在这四层上生长。下面拆三层最关键的：

      4.1 Skills · 把领域 know-how 变成可版本化的资产

      Skills 用 SKILL.md（YAML 前置元数据 + Markdown 正文）+ 可选脚本 + references/ 组织，采用渐进披露（progressive disclosure）：

      • Metadata（~50 tokens）· 只读 name + description，用来判断要不要激活；
      • SKILL.md 全文（~500 tokens）· 激活后读完整流程指南；
      • references/（2000+ tokens）· 需要时按需加载的深度文档。

      结果就是：一个 agent 可以挂几百个 skill 而不炸上下文。这也是为什么「Claude X」不需要各自造 harness——只需要造 skill 包。更关键的是，skill 不需要工程师写——PM、分析师、领域专家用 skill-creator 30 分钟就能起一个。这是把「组织隐性知识」大规模数字化的当前最优实践。

      4.2 MCP · Agent 世界的 USB-C

      MCP（Model Context Protocol）是 2024-11 由 Anthropic 开源、2025-12 被 OpenAI/Google/Meta 联合背书的事实标准。它解决的问题是：让 M 个 agent × N 个系统的连接复杂度从 O(n × m) 降到 O(n + m)——每个系统只需要写一次 MCP server，所有 agent 都能用。

      MCP 的存在直接决定了「Claude X」的可扩展性：Legal 一次上 20+ connector（Westlaw / Everlaw / iManage / DocuSign / Harvey）、Finance 上 S&P / LSEG / Moody's / Aiera、Science 上基因组 / 蛋白质数据库——全走一套协议。

      4.3 Managed Agents · 从 Claude Code harness 到「人人能造的 harness」

      Managed Agents 是 2026-06 GA 的官方托管 agent 运行时。它对 Claude X 全家桶的意义是底座的商品化——把 Anthropic 内部造 Claude X 的那套东西打包卖出去。

      五五大差异维度：拆开 Claude X 的真差异#

      — Surface · Skills · Connectors · Topology · Trust

      如果所有 Claude X 都在同一栈上，那它们的差异到底在哪？抽掉品牌包装后，只剩 5 条轴：

      两个观察：

      • 产出物形态其实是 skill 的一部分。Code 产 diff / commit，Cowork 产 doc / xlsx / pptx，Design 产 prototype / HTML / PPTX，Science 产可复现的 Notebook + lineage——这些差异都在 skill 的模板文件里，不是底座变了。
      • Finance 的「Cell-level Citations」和 Science 的「三层 Agent」是最强合规 skill。前者做「可审计」，后者用 reviewer 抓 hallucination。它们不是新架构，是同一套 Managed Agents 里的 Orchestrator 模式被强推为该垂直的默认。

      六战略动机：技术减法 · 品牌加法#

      — Why one platform × many brands

      如果技术底座只有一个，为什么还要以「Claude for X」独立品牌一次次发布？——因为技术抽象和商业叙事回答的是两个不同的问题。

      动机按重要性排序：

      1. 模型商品化，价值上移。Opus / Sonnet 迟早会被 GPT-6 / Gemini 4 打平。Skill 库 + 连接器 + 参考架构 + 客户工作流才是护城河——跟客户 SOP、模板、审批流深度耦合，切换成本极高。
      2. 拓展 TAM。开发者市场千万级 seats，加上 Cowork / Design / Finance / Science 打开的知识工作者 + 专业人士，是十亿级 seats。这是「Code 破圈后必须找到的下一片海」。
      3. 合规和数据边界是垂直不可跨越的门槛。金融要审计追溯、Gov 要保密、Science 要可复现、Legal 要权限控制。打包成「Claude for X」是进入受监管市场的通行证——单一品牌带来单一合规认证。
      4. 生态飞轮。每个垂直吸引对应的 partner：Salesforce → Finance；Thomson Reuters → Legal；Broadcom / Google / SpaceX → Science 的算力协议。Partner 带来 skill 内容，skill 带来客户，客户数据反哺模型。
      5. 反事实基准 + 数据飞轮。Finance / Legal / Science 里跑真实工作流，比通用 chat 收集到高浓度的领域评估信号——直接反哺 skill 迭代和模型 RL 对齐数据。
      6. Managed Agents 释放的第二重杠杆：以前只有 Anthropic 能造 Claude X，现在客户自己也能——即使摩根造出「Morgan Finance Agent」比 Claude Finance 更好，它也跑在 Managed Agents 上，Anthropic 依然收租。这才是「一个平台」商业上真正的雪球。
      一句话战略

      Anthropic 在做的事跟 Salesforce（一个 CRM 平台 + N 个 Cloud 品牌）的老玩法很像：技术上做减法（一个平台），品牌上做加法（N 个垂直）。差别在于 Anthropic 的底座（Model + Loop + Skills + MCP）比 Salesforce 的 Force.com 更通用、迭代更快、生态更开放。

      七FDE / ToB 映射 · 未来 Claude X 预测#

      — Three actionable moves + 5 future brands

      先说未来。Cowork Plugin 目录里已经暗示了下一波品牌，我按「合规壁垒 × partner 生态 × Anthropic 吃狗粮」三个信号做一次概率排序：

      判断规则：什么行业会被 Anthropic 亲自品牌化，什么会留给 Managed Agents 上的第三方？看三条——

      1. 合规壁垒高（金融/医疗/法律/国安）→ Anthropic 亲自造，因为要它的品牌背书才能进。
      2. 有强 partner 生态（Salesforce / Thomson Reuters / Broadcom）→ Anthropic 亲自造，因为 partner 需要一个明确的合作对象。
      3. Anthropic 自己吃狗粮（Code=工程 / Design=品牌资产 / Finance=对内财务）→ 亲自造，因为跑得通才敢卖。

      三条都不符合（零售 / 物流 / 制造业车间管理）→ 交给 Managed Agents 上的第三方造。

      八典型误读：4 条常见错读#

      — Don't confuse the skin with the platform

      陷阱 1 · 把 Claude Design 当 Figma 替代品

      Design 的核心不是画图工具，是「PM / 创始人 / 营销把想法快速可视化」的 skill 包 + Canvas surface。它不跟 Figma 争专业设计师，它跟「先做一个原型说服团队」这个动作争。误读会低估它的市场：它的对手是 PowerPoint、白板、Notion，不是 Figma。

      陷阱 2 · 把 Claude Cowork 当 Microsoft Copilot 的 Claude 版

      Copilot 是「嵌在 Office 各个 app 里的助手」；Cowork 是「独立桌面 Agent，跨所有 app / Slack / 浏览器 / 本地文件」。形态不同 → surface 不同 → 未来路径不同。Copilot 强在 Office 深度集成，Cowork 强在跨系统编排 + skill 生态。

      陷阱 3 · 把 Claude for Finance / Legal 当 SaaS

      它们不是又一个金融 SaaS 或法律 SaaS。它们是把 Managed Agents 的通用能力包装成受监管市场的合规容器。买它的机构不是买软件，是买「一个能审计、能授权、能对接 20+ 数据源的行业 agent 参考架构」。

      陷阱 4 · 把 Managed Agents 当作 Anthropic 云 / IaaS

      Managed Agents 是harness 托管 + sandbox 可选，不是通用 IaaS。它跟 AWS / Cloudflare 是互补关系——Loop 在 Anthropic 云、Sandbox 在用户侧（Cloudflare / Daytona / Modal / 自建）。把它跟 AWS Lambda 类比会错得离谱。

      九待验证 6 问#

      1. Cowork 能否像 Claude Code 一样破圈？知识工作者门槛比开发者高——UI 复杂度 / 学习曲线 / 组织 IT 阻力都比 CLI 大得多。是我们低估了它，还是它需要 12 个月才能证明？
      2. Skill 生态会不会碎片化？Anthropic 想把 Agent Skills 变成开源标准（agentskills.io），但 OpenAI / Google 会不会造自己的 skill 格式？跨平台可移植性能否守住？
      3. MCP 能否守住标准地位？OpenAI / Google / Meta 都背书了，但也都有自己的 tool-use 语义。未来 12 月是巩固还是分叉？
      4. 客户在 Managed Agents 上造的 Claude for X 会不会侵蚀 Anthropic 亲自造的品牌？比如某家银行造出「XX Bank Finance Agent」比 Claude Finance 更好，Anthropic 靠底座收租够不够抵消？
      5. 未来 HR / Sales / Health 会是 Cowork 品牌下的 plugin，还是独立品牌？时间点是 6 月、12 月还是 18 月？信号看什么——partner 发布？垂直 skill 数量突破阈值？
      6. Salesforce / 微软会不会反收割？微软 Copilot + Agent SDK / Salesforce Agentforce 都在做同样的「一个平台 + N 个垂直」。谁的 skill 生态密度先起来（尤其是被行业 SME 自己写的 skill），谁就赢——但 Anthropic 有先发优势和 MCP 标准优势。
