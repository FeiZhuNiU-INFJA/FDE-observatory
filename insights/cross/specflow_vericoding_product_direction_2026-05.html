      自然语言 → 形式化规范 → 数学证明

      2026-05-23 产品方向 Vericoding

      关联洞察：代码Review瓶颈与Vericoding范式转移

      AI产出速度 > 人类review速度，这个瓶颈已经到来。

      解决方案不是让人类review更快，而是让代码自带证明。SpecFlow从vericoding链条的上游切入——先解决"怎么写规范"的问题。

      核心主张

      当下痛点：AI代码review虽能自动化，但仍需人类审阅，无法真正解决速度不匹配

      未来方向：Vericoding（验证性编码）——代码+数学证明，无需人工review

      产品切入点：形式化规范是vericoding的上游瓶颈，也是最大的产品机会

      问题定义¶

      AI代码生成工具的普及创造了一个新的矛盾：

      写代码 review 验证
      • AI生成代码的速度已经远超人类review的速度（Faros AI 2025：PR数量+98%，review耗时+91%）
      • 即使有AI辅助review，仍需要人类最终确认——这只是延缓而非解决瓶颈
      • 真正的问题是信任：我们如何相信AI生成的代码是正确的？
      根本矛盾

      Vibe Coding（凭感觉写代码）的假设是"运行通过=正确"，这在AI时代不再成立——AI生成的代码可能看起来正确，但包含edge case bug或安全漏洞。

      Vericoding（验证性编码）的承诺是：代码附带数学证明，对于所有可能的输入都满足规范。但门槛太高——需要PhD级的形式化方法专业知识。

      核心概念解读¶

      什么是形式化规范？¶

      形式化规范 = 用数学语言写"验收标准"，让计算机证明代码永远正确

      这不是代码，是精确的约束描述。理解它的最好方式是对比：

      维度 自然语言规范（平时写的） 形式化规范
      例子 "用户余额不足时，转账应该失败" ∀s,t. 余额(s,t.from) < t.amount ⇒ 结果(s,t) = s
      歧义性 什么叫"不足"？什么叫"失败"？ 无歧义，数学语言
      验证方式 人工review/测试 工具自动证明
      保证 "这几次运行是对的" "对于所有输入，都正确"

      和单元测试的区别¶

      关键区别

      单元测试 = 采样验证。运行代码，检查特定输入下的输出。只能发现被测试覆盖的问题。

      形式化规范 = 穷尽验证。用数学方法证明代码对于所有可能的输入都满足约束，不需要运行代码。

      简单说：SpecFlow是"超纲版"的单元测试生成器——不仅生成测试，还生成"对所有输入都成立的数学证明"。

      形式化规范如何验证？¶

      验证分两步：

      1
      语法检查

      验证规范本身的格式正确（像检查JSON格式）。运行：$ dafny verify spec.dafny

      2
      逻辑验证

      验证器用符号推导（不是运行代码）证明代码满足规范。内部使用定理证明器（如Z3、CVC5）做数学推导。

      类比数学证明：你想证明"对于所有偶数n，n²也是偶数"。不是把所有偶数都算一遍，而是用符号推导：设 n = 2k，则 n² = 4k² = 2(2k²)，证毕。形式化验证器做的就是类似的事情。

      什么是Dafny？¶

      Dafny = 可以证明正确的编程语言

      由Microsoft Research开发，是学习形式化方法最友好的入门语言之一。你可以同时写：

      • 可执行的代码（像Python/Java那样）
      • 数学规范（前置/后置条件、不变式）

      一个完整的Dafny例子：

      method Abs(x: int) returns (r: int) ensures r >= 0 // 后置条件：返回值永远非负 ensures r == x || r == -x // 后置条件：返回值要么是x，要么是-x { if x < 0 { r := -x; } else { r := x; } } // 验证器会思考： // 情况1：x >= 0 → r = x → r >= 0 ✓ r == x ✓ // 情况2：x < 0 → r = -x → r >= 0 ✓ r == -x ✓ // 结论：✓ 代码满足规范

      运行验证：

      $ dafny verify Abs.dafny ✓ 验证通过

      Dafny的验证器不是运行代码，而是穷举所有逻辑路径，用数学方法证明规范成立。

      为什么SpecFlow先支持Dafny？¶

      门槛最低

      语法类似主流编程语言，不需要重新学习全新的范式

      工具成熟

      有VS Code插件、IDE支持、丰富的文档和示例

      单机场景

      最常见的业务逻辑，适用面广（82%成功率，论文数据）

      核心洞察¶

      通过调研12+篇最新论文（CLEVER、VERINA、MetaMateCR等），我们发现：

      瓶颈在上游

      规范是卡点

      VERINA显示OpenAI o3在证明生成上成功率仅4.9%。但更早的问题是：如何写出正确的形式化规范？

      自然语言不够

      NL描述无效

      Vericoding基准测试表明，添加自然语言描述对LLM生成证明无显著帮助。需要结构化的形式化规范。

      UX决定成败

      Meta的经验

      MetaMateCR发现仅向作者显示AI patches（而非reviewer）才能避免减速。谁看到什么，何时看到，是产品设计的关键。

      这些洞察指向同一个结论：

      产品机会

      形式化规范写作工具是vericoding链条上最关键、也最缺失的一环。

      解决这个痛点，既能服务当下（提升代码质量），又能面向未来（为vericoding铺路）。

      SpecFlow：产品方案¶

      产品定义¶

      SpecFlow是一个AI驱动的自然语言到形式化规范翻译器，让不会TLA+/Dafny/Lean的开发者也能写出可验证规范。

      核心能力¶

      1
      需求意图识别

      从自然语言中提取：功能需求、性能约束、安全要求、业务规则

      2
      不变式提取

      识别"永远不会..."、"必须始终..."、"确保..."等关键表达，转化为形式化不变式

      3
      边界条件分析

      从代码和需求中推断边界情况（空值、并发、异常），生成对应条款

      4
      多语言支持

      根据问题类型自动选择：TLA+（并发）、Dafny（单机）、Lean（通用）

      5
      可解释性

      AI用自然语言解释每个形式化条款的含义，标注可疑点

      风险与缓解¶

      风险1：自然语言歧义性

      问题：NL无法完全消除歧义，可能导致错误规范

      缓解：(1) 引导式输入模板 (2) AI追问澄清 (3) 可疑点强制人工确认

      风险2：形式化工具生态碎片

      问题：TLA+/Dafny/Lean各有生态，兼容成本高

      缓解：(1) 内部中间表示 (2) 先聚焦Dafny（门槛最低）(3) 社区集成插件

      风险3：教育曲线

      问题：即使有工具，用户仍需理解形式化规范的基本概念

      缓解：(1) 渐进式学习路径 (2) AI解释作为教育工具 (3) 丰富的示例库

      为什么是现在¶

      技术就绪

      LLM在代码生成和翻译任务上已足够成熟；vericoding基准显示可行性（Dafny 82%成功率）

      市场痛点

      AI代码review虽火热，但只是"止痛药"而非"解药"。vericoding才是终极方案

      时间窗口

      形式化方法工具正在从学术走向工业。现在切入可抢占"vericoding工具"品类定义权
