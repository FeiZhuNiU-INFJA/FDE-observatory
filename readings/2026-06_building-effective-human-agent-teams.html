      核心结论：过去人用 AI 是「单人游戏」——一个人对一个聊天窗口。当 Agent 有了持久记忆、独立凭证、跨团队上下文之后，工作变成「多人游戏」：一队人定战略，Claude 执行。让这局游戏赢下来的，不是模型更大，而是四条团队准则——信息公开、角色明确、设北极星、逐步授权。Anthropic 在自家团队跑了几个月，把它们做成心法：Agent 是新同事，评估、验收、扩权和给人一样的节奏；差别只在于「你有多少信息落到文档、多少工具向它开放」。
      一页要点
      • 范式转移：AI 从「1 人 × 1 chat」的单人玩法，进化为「团队 × 多人 Agent」的多人玩法——工作从任务级迁到目标级
      • 多人 Agent 的三大能力：持久记忆、不与人绑的独立凭证、持续宽范围上下文访问——三者缺一，Agent 进不了团队
      • Lesson 1 · 公开工作：Agent 只认「被写下且可搜」的东西——把安全边界从「按文档」升到「按工作区」；对 Agent 而言，没写下 = 不存在
      • Lesson 2 · 明确角色：一个人一个 Agent 一份职责；同一线程共享工件——避免每人另开一支 side agent 分裂上下文
      • Lesson 3 · 设北极星：写下一个团队级远大目标 + 明确指定哪些 Agent 可以「主动提议新工作」——最好的 Agent 不是接活的，是提活的
      • Lesson 4 · 逐步建立信任：像招新人一样评估 → 反馈 → 扩权；Doer-Verifier 双 Agent harness + 每周「lessons & missteps」自省报告；把人当稀缺资源，Agent 要学会打包问题、给足上下文
      • 对 FDE 的意义：客户组织能不能用好 Agent，不看模型，看「信息公共度 + 角色书面化 + 目标外化 + 反馈闭环」四条——这是我们卖服务的抓手

      一范式转移：从单人游戏到多人游戏#

      The Shift · single-player → multiplayer

      过去两年，人和 AI 的工作方式是「单人游戏」：一个人开一个聊天窗，把 Agent 当私人助理，处理编码、研究、财务分析——工具从 IDE 换到表格再到 slide，但底色始终是「1 人 × 1 Agent × 1 任务」。Claude Tag 出现后，人和 Agent 处在同一个协作空间（如 Slack、共享文档），围绕团队共同目标协作。工作从「个人任务」升维成「共享目标」，看起来更像一局多人游戏——人定战略，Claude 执行。

      私聊窗 · 一次一件事 · Agent 无跨会话记忆、无团队凭证、无广域上下文。任务级协作。

      now · 多人游戏

      团队 × 多 AgentMultiplayer

      Slack / 共同文档 · 同一 roster · 目标级协作。Agent 有自己的凭证、记忆和角色。

      二多人 Agent 的三大能力：进团队的前置门槛#

      Three Capabilities · what makes an agent multiplayer

      要让 Agent 在团队频道里「有产出地参与」，不是把聊天机器人塞进 Slack 那么简单。Anthropic 给它三条硬能力——它们是 Agent 从「个人助理」进化到「团队同事」的门槛：

      01

      持久记忆Persistent Memory

      记住团队目标、之前的决策、跨会话的偏好——才能把每次执行都调向目标，而不是每次都从零开始。

      02

      独立凭证Own Credentials

      Agent 有自己的账号 / token，不再借用某个人的身份——这样才能进入安全、可预测的护栏，权限归属清晰。

      03

      广域上下文Broad Access

      持续访问 Slack、代码、文档、会议纪要——学会「组织怎么运转」，才能执行服务于团队目标的任务。

      三条只是入场券

      技术能力齐备只解决「能进团队」；要让人-Agent 团队真正「跑得赢」，还需要团队级的工作方式与共识——这就是后文四条 Lesson 的意义。

      三四条 Lesson：让多人 Agent 团队跑起来的心法#

      Four Lessons · the operating principles

      Anthropic 内部试了几个月，沉淀出四条准则——它们互相支撑，缺一条就会漏水：

      Lesson 01

      公开工作

      Work in Public

      给 Agent 广阔上下文——信息默认公开、按工作区划安全边界，不再按文档一份份放。没被写下的东西，对 Agent 就不存在。

      → 减少「能不能分享」的决策疲劳
      Lesson 02

      角色明确

      Defined Roles

      每个 Agent 有一份技能文件、一份可访问工具、一段职责描述。人和 Agent在同一线程工作，任何人都能接手。

      → 避免团队里悄悄跑一堆 side agent
      Lesson 03

      设北极星

      North Star

      一个远大团队目标，写下来，共享给 Agent；明确指定哪些 Agent 可以主动提议新工作流。

      → 从「接活的 Agent」升到「提活的 Agent」
      Lesson 04

      逐步授权

      Build Trust

      和招新人一样：评估 → 反馈 → 扩权。Doer-Verifier 双 Agent 交叉验证 + 每周「lessons & missteps」自省报告。

      → 单 Agent 独立完成 500 bug fixes 的路径

      3.1 Lesson 1 深读 · 公开工作 = 给 Agent 广阔上下文

      Agent 只吃「已写下且可搜索」的东西：Slack、代码、文档、会议纪要。私聊消息、走廊闲聊、限权文档，对它一概不存在。Anthropic 的做法不是「一份份文档批 Agent 权限」，而是把安全边界升到工作区级——一个工作区内、人和 Agent 都是队友。

      公开工作的三个动作
      1. 选出组织的几个「安全边界」（如 R&D 内部 / 客户信息 / 财务），把工作区与分享设置和边界对齐；
      2. 新频道默认公开；决策落到频道、文档、纪要里——不落笔就等于没发生；
      3. 把 Agent 也纳入产物（artifact / 会议纪要）的写作预期——它现在是文档的头号读者。

      为什么值得：能读到会议决定的 Agent 不会再提已被否掉的项目；能读到别人 spec 的 Agent 会推荐别的团队验证过的模式；因为它能瞬时吞吐大量文本，它经常翻出人错过的相关工作。敏感事务仍然可以用 DM / Claude.ai / Cowork 走私域通道。

      3.2 Lesson 2 深读 · 角色 = 工具 × 记忆 × 边界

      人和 Agent 共享一份 roster、一套产物、一个工作空间。不同 Agent 持不同角色：一个负责数据分析、一个把关设计规范、一个做研究综合。项目启动时，人和 Agent 一起商量谁做什么。一旦分工确定，某个 Agent 会再拉起子 Agent——比如数据分析 Agent 要拿 BigQuery、QA Agent 要拿 Playwright MCP。

      反面：没有角色会发生什么

      「每人在旁边偷偷跑一支个人 AI」——工作重复、上下文分裂。经典例子是指标看板：多人 Agent 干一次，全团队看到同一份数字；side agent 各干一次，就出现各版本的数字。

      Anthropic 一个工程团队开始为「人 + Agent」都做roster——每个岗位配 skill file，角色和职责被写下来后，扩起来快多了。例如项目复杂到需要新一层协作，他们直接加了个 release manager agent专门处理发布。这就让「多 Agent 团队」在心智上可以像多人团队一样自然生长。

      3.3 Lesson 3 深读 · 北极星 = 让 Agent 主动提议的许可证

      Anthropic 里最重要的 Agent 不是「接了活干」，而是「主动提议新工作」。触发条件是：团队已经给了它丰富上下文和明确角色，然后再加一层——北极星。北极星总是由人来设、扎根在业务与使命上；然后明确指定哪些 Agent 有资格主动提议——不是每个 Agent 都有这个能力和信用。

      案例：onboarding 团队

      一个内部工具团队的北极星是「让 onboarding 更好用」。某个 Agent 主动提议了 onboarding 报错文案的一组改写。改上线后，onboarding 成功率下一周就可测量地提升。人没派它去改文案；是它读着北极星自己动的手。

      3.4 Lesson 4 深读 · 信任是一条曲线，不是一个开关

      「工程师能派 Agent 独立跑完 500 个 bug fix」听起来很猛，但一开始不是这样。授权范围与可靠度成正比——就像评估一个新同事：先手工看每一次决策 → 教它 surface 那些真需要人的岔路 → 让它自己带一个「lessons & missteps」的自省报告 → 逐类任务扩权。

      ← 起步：全审 ▲ 反馈闭环 · 每次外推一类任务 成熟：独立 dispatch →

      关键点是：Agent 的所有产出都要能被验证——代码有测试，技术文档有 rubric 和 style guide。「让 A 干活、让 B 检查」这种双 Agent 组合，Anthropic 叫它 Doer-Verifier harness。

      Doer-Verifier harness：一个 Agent 干活、另一个 Agent 检查——只有过了 Verifier 才递到人手里。Anthropic 的一个工程 leader 用这套让 Agent 处理积压：先一批 Agent 打分与派单，另一批吃掉中低复杂度 issue；每周 Agent 自己写「lessons & missteps」回顾。几周下来，leader 花在 day-to-day 指挥上的时间越来越少。

      把「人的注意力」当稀缺资源

      Agent 独立之后，教它们珍惜人的注意力：批处理问题一次问完、重复关键上下文让人快速上手、限制同时抛给一个人的事情数量。有的团队甚至安排一个专职 Agent，只负责决定什么该向人升级。也可以反向设护栏：限定 Agent 每天做的工作量，保证人能有意义地参与、不过滤成路人。

      四FDE 映射 · 落到我们的现场#

      Field Mapping · 科沃斯 / 添可 / SeedAgent

      1. i「组织公共度」是新 SLO：客户能不能用好 Agent，第一杠杆不是模型，而是信息有没有落到 Agent 能读的地方。评估科沃斯电商客服、添可社媒团队时，加一条：私聊 / 群 / 走廊里流转的决策比例——比例越高，Agent 上限越低。
      2. ii用 roster 谈单，不用「Agent 数量」谈单：把「每个 Agent 的 skill file + 工具权限 + 职责」写成客户可以签字的roster 表。参照添可视频质检：Brief 提取 Agent × 质检 Agent × 脚本反向生成 Agent × 投放诊断 Agent，同一线程共享工件——比讨论「你要几个 Agent」有效得多。
      3. iiiSeedAgent 需要一个北极星：SeedAgent 走 RSI 架构（Lilian Weng 的 harness engineering），如果只有「做完当前任务」这样的局部目标，永远长不出会「主动提议改进」的 Agent。北极星越具体（如「让 impact-per-token 每季 ×1.3」），主动提议越有落点。
      4. ivDoer-Verifier 是电商客服的必选项：Thinking 模式把售前转化提 11.6%、售后提 5.55%——如果再叠一个专职 Verifier Agent（校对话术、拦截合规红线），能把「值不值得看这条对话」的判断成本降到接近零，让运营真的敢抽样式管理而不是全审。
      5. v信任曲线 = 交付节奏表：给客户 POC 的第一阶段就锁死「先手动核 → Verifier 上岗 → 按任务类型扩权」三档；把「每周 lessons & missteps 回顾」写进交付节奏。这条比任何模型 benchmark 都更容易让客户签第二期。
      判断

      这篇是 Anthropic 少见的「组织层」文章——没讲模型、没讲 harness 结构，讲的是「人怎么和 Agent 组队」。它的份量在于：把「多人 Agent 到底是什么」定义清楚了（三大能力 = 门槛），把「怎么让它跑赢」拆成了四条可复制的团队准则。它的边界是：结论主要来自 Anthropic 内部（工程 / 内部工具团队），跨行业迁移到科沃斯家用机器人这种客户侧还需一层适配——尤其是「公开工作」在ToC 用户数据面前有天然的合规边界。

      五待验证问题#

      Open Threads

      1. 「公开工作」在中国企业（KPI/OKR 文化 + 强层级）里如何本土化？"decision fatigue" 之外还有「政治成本」的信息公开阻力。
      2. Doer-Verifier 的 Verifier 要不要用更强的模型？还是同模型、不同 skill file 就够？——这直接影响成本。
      3. 「主动提议新工作」的 Agent 如何在企业内避免提议疲劳（提太多、提偏了）？是否需要专门的「提议 → 排序 → 归档」子流程？
      4. 北极星和 OKR 的关系：北极星更长、更粗；OKR 更短、更细——两者要不要同层维护，还是 Agent 只看北极星？
      5. 500 bug fixes 之后是什么？Anthropic 显然还没给答案——但它决定了「人-Agent 团队」的边界。
