      一背景：从手动提示到循环工程#

      The Shift · 范式转移

      维度内容
      视频来源Best Partners TV · 基于 Google AI 总监 Addy Osmani 深度分析 · 约 18 分钟
      核心信源Addy Osmani X 帖；Peter Steinberger（OpenClaw）；Boris Cherny（Claude Code）；Andrej Karpathy（AutoResearch）
      概念层级Agent Harness Engineering（单 Agent 运行框架）< Loop Engineering（计时器上的自主运行环境，可生成子 Agent）< 工厂模型（整套软件构建系统）
      过去 · 手动提示

      Agent 是「你全程握持的工具」——每一步都要人触发，串行、走走停停，产能上限是你的手速。

      → 范式转移
      现在 · 循环工程

      你只定义最终目的，循环在后台反复迭代执行；人从「逐步发令」变成设计能自转的系统。

      Boris Cherny：「我现在很少手动提示 Claude 了，有很多循环在后台运行，它们负责提示 Claude、判断下一步。我的核心工作是编写这些循环。」

      二循环工程是什么#

      Definition · 概念定位

      Loop Engineering（循环工程）：用你设计的系统，替代你自己完成对 Agent 的提示与调度。循环 = 一个递归目标——你只定义最终目的，AI 反复迭代执行直到完成。

      最外层工厂模型 Factory Model

      整套软件生产系统——从需求到交付的完整流水线。

      本文焦点Loop Engineering 循环工程

      跑在计时器上，能自主生成子 Agent、自我驱动、持续运转。

      最内层Agent Harness 运行框架

      为单个 Agent 搭建运行环境（prompt、工具、权限）。

      关键变化：一年前跑自动循环还需自己写 bash 脚本长期维护；现在五模块能力已内置进 Claude Code 与 Codex，Steinberger 总结的循环清单与两款产品功能几乎一一对应。启示是：设计通用循环逻辑，工具可替换。

      三五模块 + 外部记忆#

      The Loop Engine · 循环引擎

      1Automations 自动化 —— 给任务设周期，到点自动触发，循环的心跳。
      2Worktrees 工作树 —— 多 Agent 并行改码的物理隔离，避免撞库。
      3Skills 技能 —— 写一次、每次可读，消除意图债务，意图复利。
      4Connectors 连接器 —— 基于 MCP，读需求/查库/建 MR/发通知，融入环境。
      5Sub-agents 子 Agent —— 第二个模型生成分离验证，发现自审盲区。
      ⏚外部记忆 —— 跨会话状态。Agent 会忘，仓库和状态文件不会。

      外部记忆External Memory · 跨会话状态

      Markdown 状态文件、项目看板——记录「已完成/待做」。长时间运行的循环必须依赖持久化存储，Agent 会忘，仓库不会。

      基座 / Foundation

      3.1 Automations —— 定时与持续运行

      两款工具实现路径不同、逻辑相同：

      能力CodexClaude Code
      定时任务自动化标签页：选项目、提示词、频率、本地/工作树/loop、定时任务、生命周期钩子（hooks）
      持续运行至条件达成/goal：跨多轮直到可验证停止条件成立/goal：每轮结束由独立小模型检查是否完成
      无人值守结果进分类收件箱；无问题则自动归档可推送到 GitHub Actions 关电脑后继续
      Skill 调用自动化任务可直接调用 Skill同上逻辑

      /loop = 固定节奏重复；/goal = 直到「认证模块测试全过且 lint 通过」类停止条件成立。写代码的 Agent 与判断是否完成的 Agent 不是同一个——生成与校验分离从停止条件层就已体现。

      3.2 Worktrees —— 并行隔离

      Codex 内置多线程；Claude Code 支持 --worktree 与子 Agent 工作树隔离，多 Agent 同时改同一文件从物理层避免冲突。

      编排税

      工作树只解决机械层文件冲突。真正瓶颈是人一天能审核多少份产出——你能并行跑几个 Agent，取决于你能读几份 PR，而非工具线程数。

      3.3 Skills —— 意图复利

      每次新会话从零解释项目 → 意图债务：没说清楚的，Agent 用猜测填补。Skill 写一次、每次可读。

      • Skill = 内容编写格式；Plugin = 内容分发方式（跨仓库共享、打包多个 Skill）
      • 描述要简洁明确——准确匹配比花哨文案有用

      3.4 Connectors —— 融入工作环境

      基于 MCP 协议的连接器让循环读需求跟踪器、查数据库、调测试环境、发 IM，两款工具 MCP 互通可跨工具复用。普通 Agent 告诉你「这里有修复方案」；完整循环能自己建 MR、关联工单、CI 通过后自动通知频道。

      3.5 Sub-agents —— 生成分离

      第二个 Agent（不同指令/模型）能发现第一个忽略的问题。常见分工：探索 → 实现 → 对照规格验证。无人值守时，可信验证环节是放心让它跑的前提；代价是更多 token，只在关键场景开启。

      四完整循环实例#

      Loop in Motion · 每日 CI 巡检

      你只需设计一次循环规则，之后无需手动提示任何步骤——这正是 Steinberger「不要手动提示 Coding Agent」的工程化体现。

      1触发
      Automations

      早上自动启动，调分类 Skill，读昨日 CI 失败 / 未解 issue / 最近提交。

      2落盘
      Memory

      写入状态文件（Markdown / 看板），建立可断点续跑的基线。

      3分支
      Worktree + Sub-agent

      每个问题建工作树：子 Agent A 起草修复，B 对照 Skill/测试审查。

      4交付
      Connectors

      连接器自动建 MR、关联工单；CI 通过后通知频道；复杂问题进人工收件箱。

      5续跑
      Memory

      状态文件记录已尝试方案 / 通过验证 / 进行中，次日从断点继续。

      ↻ 次日重复 · 从断点继续，而非从头开始

      五三笔「债」：循环放大的副作用#

      The New Debts · 递进的风险

      循环让产出变快，也把三类风险同步放大——严重度自左至右递增：

      Lv 1

      验证责任Verification Responsibility

      无人值守循环 = 无人值守犯错。「完成」只是一句声明，不是严格验证。

      应对交付的仍是你亲自确认可运行的代码。

      Lv 2

      理解债Comprehension Debt

      循环产出越快，你没亲手写的代码越多，对代码库的理解持续退化。

      应对认真读每一份循环生成的代码，保持掌控力。

      Lv 3

      认知投降Cognitive Surrender

      循环自运行时，人停止主动思考，直接接受所有结果——加速能力退化。

      应对带着判断力设计循环；逃避思考则是最深的债。

      此外还有现实约束：token 成本（不同模式消耗差异巨大）、代码质量下滑（无人值守放大粗糙产出）。方向仍处早期，但可能是未来协作雏形。

      六FDE / ToB 映射#

      Field Mapping · 落到客户现场

      1. i杠杆点转移：从「写好单次 prompt」→「设计可复用循环系统 + 组织记忆」——与 Alex Albert 访谈中「eval / 书面文化 / 并行 Agent 治理」同向。
      2. ii工具选型降级：五模块同构意味着客户场景应优先问「循环逻辑是否可审计、可回滚」，而非「Claude Code vs Codex」。
      3. iii编排税 = 交付瓶颈：ToB POC 并行 Agent 时，审核/验收人力是硬上限；循环设计必须包含人工收件箱 + 分级自动化。
      4. ivSkill = 组织意图资产：对应企业内 coding 规范、架构决策记录、踩坑清单——循环每次运行都复利读取。
      5. v子 Agent 验证 = eval 治理：生成分离、对抗审查，是无人值守场景最低限度的质量门禁。
      判断

      循环工程是解药也是毒药——取决于你是否仍承担验证与理解责任。可以搭建循环，但要以工程师身份设计系统，而非做只会按启动键的人。找到「直接提示」与「自动化循环」之间的平衡，比全盘押注任一极端更务实。
