文档 01 · 完整调研

多智能体算力调度
独立调研报告

基于 2024–2026 公开论文与开源项目，梳理 OS 派、Workflow-Aware 派、基础设施派与经典资源分配四条技术路线。

📅 2026-06-08🔬 公开论文 + 开源项目

摘要

🎯 核心发现

1. L1/L2 已有较成熟方案（MLFQ、PLAS、load function 等），但多针对公平性与 HoL blocking，缺乏业务语义。

2. 2025–2026 范式正从 request-level 转向 workflow-level（SAGA、Agentix、TokenCake），整段 Agent 工作流成为一等调度单元。

3. L2↔L4 贯通（业务 SLA / 租户 / 用户打断 → 可执行优先级）仍是明显空白。

1. 问题定义

1.1 核心问题

在任意时刻，哪个 Agent（或哪次 LLM 调用）应优先获得资源？
资源类型池化形态争用表现
GPU集群 N 卡、fractional GPU显存碎片、KV cache 驱逐、冷启动
API 并发provider rate limit、自建 lane 池429 级联、HoL blocking、zombie
Token 预算组织/租户/任务级 cap长任务挤占、批处理饿死交互
CPU/内存orchestrator 进程池tool 等待时空转、编排阻塞

1.2 四层问题模型

L4业务优先级SLA、租户 tier、人工审批、用户打断
L3工作流编排DAG 关键路径、动态 replan、subagent 依赖
L2推理槽位并发 lane、MLFQ、模型路由、admission control
L1硬件池GPU 分片、SM 时间配额、API quota
层级调度对象典型瓶颈代表方案
L4 业务SLA、租户、审批付费 vs 批处理、紧急打断各平台策略层（多数未产品化）
L3 工作流subagent、DAG 依赖关键路径阻塞、tool 等待空转SAGA、TokenCake、FATE、HexAGenT
L2 推理槽位并发 LLM 调用HoL blocking、zombie、429AgentRM、Agentix、HiveMind、loco-agent
L1 硬件池GPU/CPU/API quota显存碎片、冷启动、SM 争用Ray Serve、KAI Scheduler、HAS-GPU
⚠️ 常见误区

把问题简化为「调大 concurrency」或「加更多 GPU」。低负载下看似够用；当 N 个 Agent 争用 C 个 lane（C ≪ N）时，scheduling failure 才暴露。

2. 方案全景

2.1 OS 派：把 Agent 当 Process / Thread

OS

AgentRM arXiv:2603.13110, 2026

定位Agent 框架与 Model API 之间的中间件
机制MLFQ、zombie reaping、rate-limit-aware admission、AIMD 退避
实证OpenClaw、AutoGen、CrewAI、LangGraph、Codex、Claude Code 共 4 万+ issues
效果P95 延迟 -86%，吞吐 +168%，zombie 29→0，lane waste -96%
局限无 DAG 关键路径；无业务语义

MLFQ 队列：Queue 0 高优先级 → Queue 1–2 随时长降级 → Priority boosting 防饥饿 → Zombie reaping 释放死 lane

OS

Agentix / Autellix NSDI 2026

PLASProgram-Level Attained Service：累计服务时间少的程序优先
ATLAS多线程/DAG 按 critical path 上的 attained service 排序
效果同延迟下程序吞吐 4–15×（相对 vLLM）
局限面向 serving 层；非 framework-agnostic；无多租户 SLA
🔑 关键区别

MLFQ 解决 call-level HoL blocking；PLAS/ATLAS 进一步解决 program-level HoL blocking。

OS

HiveMind arXiv:2604.17111

定位透明 HTTP 代理，零改 Agent 代码
原语admission、rate-limit tracking、AIMD、circuit breaker、token budget、priority queue + DAG
效果5–50 并发：未协调失败率 72–100% → 0–18%
OS

loco-agent 开源, PyPI 2026

公式L(i) = α·(Qi/maxQ) + (1-α)·(Dmax_i/maxDmax)
能力bounded concurrency、backpressure、SLO error budget、deadlock-safe
局限动态 replan 弱；无 KV cache 感知
📋 OS 派小结

MLFQ/PLAS/LAS 解决公平性与 HoL blocking；loco/HiveMind 解决框架无关接入；均未充分解决 业务优先级 + 动态 DAG。

2.2 Workflow-Aware 派：整段工作流为一等公民

2025–2026 最重要范式转变：不再把每次 LLM 调用当独立 request。

WF

SAGA arXiv:2605.00528 · GitHub

  • 主张：tool call 间丢弃 KV cache 导致 3–8× 延迟膨胀
  • 机制：Agent Execution Graph（KV 复用预测）、session-affinity batching、Agent Fair Share
  • 效果：64-GPU 集群 TCT -1.64×，SLO 99.2%
WF

TokenCake arXiv:2510.18586

  • Temporal：tool call 期间 offload KV，预测性 prefetch
  • Spatial：GPU 内存分 shared + reserved pool，保护 critical-path cache
WF

Chimera arXiv:2603.22206

  • 语义 router + 剩余 token 预测器 → STJF（Shortest Total Job First）+ aging 防饥饿
WF

HexAGenT arXiv:2605.16637

  • Prefill-Decode 分离；在线揭示 DAG；按「违反 workflow horizon 风险」排序
WF

FATE arXiv:2605.07238

  • future-state-aware：评分候选 assignment 对下游状态影响；CP-SAT 求解
📋 Workflow 派小结

学术前沿在 KV cache + DAG + makespan；工程落地多绑定 vLLM/Ray，与 L2 lane、L4 业务优先级尚未整合。

方案机制调度粒度Agent 相关性

2.4 经典多 Agent 资源分配

  • 拍卖/市场：去中心化、可证 welfare；对 LLM：utility 难实时估计

  • MARL：动态自适应；非平稳、reward 延迟、不可解释；建议仅在线调参

📌 结论

主流框架把调度留给用户，中间件层窗口仍在。

3. 工作负载场景

W1

Lane 争用（基础）

W2

DAG 关键路径

验证 L3 信号是否缩短 workflow makespan。

[Planner] → [Researcher_A] ──┐
     └──→ [Researcher_B] ────┤→ [Writer] → [Reviewer]
critical path: Planner → Researcher_A → Writer → Reviewer
W3

Tool 等待空转

模拟 Agent 在 tool call 期间不占 lane，但 workflow 仍存活。Tool wait 占总时间 30–70%。

W4

优先级插入（L4）

注入 priority_class=0 紧急 Agent，测 SLO 与饥饿平衡。

SLO_attainment = 1 - (violations / total_priority_0_calls)
W5

Rate Limit 级联

测 429 处理与 zombie 治理。指标：429 次数、zombie 数、lane waste。

W6

多租户公平性

3–10 租户不等 quota/tier/负载。公平性：

AFS_deviation(i) = |TCT_i / fair_share_i - 1|

文档 03 · 研究机会分析

空白区深度分析

聚焦三个尚未被充分解决的空白区：L2↔L4 贯通、多租户可审计调度、路由与 lane 联合优化。

📅 2026-06-08🎯 5 空白区 · 3 重点

1. 空白区总览

ID空白区仍缺什么难度差异化
G1L2↔L4 贯通业务语义 → 可执行优先级中高
G2多租户可审计决策可回放、合规解释中高
G3路由+lane 联合占槽与选模一体优化中高中高
G4replan 抢占关键路径变化时的让路高中

2. G1：L2↔L4 贯通

L4: "这个审批流是 P0，用户刚 inject 了新指令"
↓ （多数系统无标准接口）
L2: "按 FIFO / MLFQ 排队，不知道业务含义"
方案L2 能力L4 缺口
AgentRM执行时长反馈无 priority_class 注入
PLASworkflow 累计服务无租户/SLA override
loco-agentSLO error budget偏单 Agent，非跨 workflow 业务规则
HiveMindpriority queuepriority 需手动设置，无策略引擎

技术路径

路径 A：Priority Synthesis Layer

L2 前 Policy Engine，将 L4 规则编译为 ScheduleHints：

hints = PolicyEngine.evaluate(
  tenant_id, workflow_id, agent_id,
  event={type: "user_inject", ...},
  context={sla_tier, approval_state, est_value}
) → priority_class, preemption_eligible, max_wait_ms

路径 B：双队列 + override

  • System queue：MLFQ/PLAS 处理默认争用
  • Express queue：P0/P1 有独立 lane 配额（reserved 1 槽）
  • Override：user_inject 提升至 express，不 kill in-flight tool

路径 C：事件驱动优先级衰减

  • 插入时 P0 → 每等待 Δt 未 grant 自动升级（防饿死）→ grant 后按规则衰减

验证标准

场景通过标准
W4 优先级插入P0 SLO ≥ 95%，P2 饥饿 < 5%
混合 tier高 tier makespan P95 低于低 tier ≥ 30%
inject 延迟inject 到首次 grant < 2× 稳态 P50

3. G2：多租户可审计调度

「为什么在 14:32:07 让 Tenant A 先获得了槽位，而不是 Tenant B？」

不只是 fairness，是 合规与纠纷仲裁 需求。

路径 A：Decision Log as First-Class Artifact

grant_id, ts, tenant_id, workflow_id, agent_id, call_id,
policy_version, candidates[{id, score, factors}],
selected_reason, lane_id, preemption=false

支持 Replay（确定性重放）+ Counterfactual（「若 B 优先会怎样」）

路径 B：租户隔离 + 组内调度

TenantQuotaManager → Tenant A: max_lanes=2
                    → Tenant B: max_lanes=1
IntraTenantScheduler → PLAS / loco within tenant

跨租户争用只在 spare capacity 上发生。

路径 C：SLA 合同对象

sla_id: enterprise-gold
rules:
  - metric: p95_wait_ms
    threshold: 5000
    priority_class: 1

4. G3：路由 + lane 联合优化

在 lane 稀缺时，「降级到 Haiku」可能反而 延长 makespan（更多轮次）或 浪费槽位。
min α·E[makespan] + β·E[cost] + γ·P(SLO violation)
s.t. Σ active_calls(t) ≤ C, tokens(c,m) ≤ budget

路径 A：两阶段启发式

  1. Admission：拒绝低 outcome_value / est_cost 的 call
  2. Routing：选 cost-quality Pareto 最优模型
  3. Feedback：失败 → 升级模型权重

路径 B：Lane-aware router

router(call, state={
  free_lanes: 1,
  queue_depth_by_tier: {...},
  tenant_budget_remaining: ...
}) → model, defer=false

路径 C：Critical path 禁止降级

  • on_critical_path=true → 仅允许 ≥ 阈值模型
  • 非关键路径 → Haiku 省 budget，lane 给关键路径

5. G4/G5 简述（中长期）

G4：运行时 replan 抢占

  • Checkpoint 边界安全抢占（不中断 in-flight tool）
  • Priority inheritance：高优先级 workflow 临时提升依赖 subagent
  • 验证：makespan 劣化 < 1.2× 静态上界

G5：KV + lane 统一

  • 同一 workflow KV 驻留 GPU 与 lane grant 同步
  • 工具等待期 TTL 与 lane 释放联动
  • 验证：需真实 GPU；TCT ≥ 1.3× 分离优化

6. 开放命题：AACS

在有限 lane/GPU/API 池下，联合优化 workflow makespan、$/outcome、SLO 违约率；决策 可解释、可回放；L4 业务规则标准注入 L2。
