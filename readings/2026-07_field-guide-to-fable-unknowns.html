      源：Thariq Shihipar · Field Guide to Fable: Finding Your Unknowns（推文首发） · 附 keynote AI Engineer World's Fair 2026 · 综述 Latent Space AINews

      核心论点：模型越强、越 agentic，产出质量的瓶颈就越不是模型的能力，而是你对「自己不知道什么」的澄清程度。Fable 这一代前沿模型已经强到能替你独自穿越整片领土——它必然会走到你地图上没画的地方。把 unknowns 前置显影，比在 token 烧完之后再改 prompt 划算一个数量级。
      一页要点
      • 地图不是领土。 你的 prompt/skill/spec 是地图；代码库、需求、现实是领土。Gap 里全是 unknowns。
      • Fable 一代的自主性把你逼进整片领土。 它烧 token 快，走错的成本是不可逆的——所以要在实现前先探探路。
      • Rumsfeld 四象限里，出事的是右下两格：「unknown knowns」（太理所当然没写但一看就认）与「unknown unknowns」（压根没考虑到）。
      • 六件前置显影工具：Blindspot pass、Brainstorm、Interview me、Use references、implementation-notes.md、Quiz me——都是把不可见格提前撬开的杠杆。
      • 与 unhobbling 是一件事的两面。 Unhobbling 清「过时的 known」（例子太多、"别做 X" 太多），Unknowns 清「还没意识到的未知」。这一代新模型两者都要做一遍。
      • 对我们：把这套变成 SeedAgent / 客服 Agent / 添可视频质检的 skill 前置动作——「先澄清、再动手」的 impact-per-token 比事后返工高得多。

      一为什么这条推现在值得读#

      — WHY NOW

      Anthropic 6 月 9 日发布 Fable 5，6 月 12 日被商务部叫停，6 月 30 日解禁、7 月 1 日复售。Anthropic Claude Code 团队的 Thariq Shihipar（@trq212） 手里本来在写一整套 "Field Guide to Fable" 博客系列，正好赶上 AI Engineer World's Fair 2026 的一个主 keynote 席位——他用一晚的时间把 keynote 全部改成 Fable 上手指南，7 月 6 日发出，同期把系列第二篇《Finding Your Unknowns》挂到 X 上。

      整场 keynote 分四段：Unhobbling Claude（解开模型的脚镣） → Finding Your Unknowns（找出你的未知） → Dealing with the Grief（应对生产力跃迁的心理落差） → Being Unreasonable（要求好、快、便宜同时兑现）。这条推承载的是第二段，也是被社区二次引用得最凶的一段——因为它把「新一代前沿模型上手」的那个笼统直觉，翻译成了六个可以立刻拿去用的动作。

      语境提示：Thariq 本人的分工是 Anthropic 的 Claude Code 团队，日常就是「把 Claude 装进 agentic harness」的一线工程师。他给出的手感不是学术风的抽象化，而是「今天在生产上跑」的手工艺。这也是为什么社区已经出现 finding-unknowns-skills 这种把它复刻成 SKILL.md 的开源仓——本质上是我们这类"skill 化"团队最容易吃到的一口红利。

      二地图不是领土：为什么你的 prompt 成了产出的瓶颈#

      — MAP ≠ TERRITORY

      Thariq 借 Alfred Korzybski 的老话："the map is not the territory"——你画在纸上的东西不是那片土地。落到 Fable 的日常里，两侧的意象是完全能对上号的：

      你的地图 · The Map

      已知路径清晰、坐标可标 Prompts · Skills · Spec

      你在纸上写下的一切：需求、约束、示例、目录、"别做 X"。它是干净、有边界、可复读的——但也有限。

      → Gap = Unknowns
      真实的领土 · The Territory

      还在生长的地貌 · 有雾区 Codebase · Reality · Users

      代码库的真实结构、隐性约定、上游依赖、用户真实工作流、意料之外的边界条件——它是连续的、有雾的、随时刷新的。

      Fable 一代新模型的特征是：它的自主性大到能把这片领土一个人走完——它会掀山、渡河、闯进你没画的雾区。当它每一次「走到地图边缘」的时候，如果你的地图里没画那个点，它就得替你替下一个决定，而它替你替下的那个决定，往往就是最后事故的位置。

      推文里 Thariq 的原话："Fable 是我第一个真正逼我去找出自己 unknowns 的模型。" —— 因为在弱一点的模型上，你只需要把 known 写更细就够了；在 Fable 上，任何 unknown 都会被走到。

      三Rumsfeld 四象限：真正咬人的是右下两格#

      — WHERE THE BUGS HIDE

      他把老 Rumsfeld 的 2×2 拿过来了，但换了坐标含义：横轴 = 我是否意识到，纵轴 = 该知识是否已被表达。

      我意识到 · Aware
      我没意识到 · Unaware
      已表达 · Articulated
      未表达 · Latent
      正常
      Known Knowns我知道，且我写进了 prompt

      "用 TypeScript"、"缓存 5 分钟"、"函数名驼峰"。这一格已经在你的 spec/prompt/skill 里，模型能看到。

      → 不必额外处理

      TODO
      Known Unknowns我知道我还不知道

      "缓存 key 该怎么设计？"、"要不要支持多租户？" 这些是坦白的 TODO，你会主动问、主动调研、或者延后。

      → 老式 clarification 就够了

      隐性
      Unknown Knowns太理所当然，我没写

      "当然要有 loading 态"、"当然要防止负数金额"、"当然遵循我们仓库的日志前缀"——你一看就认，但你没写在 prompt 里。

      ⚠ 模型不会自动补出这些

      雷区
      Unknown Unknowns我根本没考虑过

      上游服务返回超时怎么办？两个协作 agent 撞了同一把锁怎么办？用户搜索里出现 emoji 会不会崩？——这些你根本没想到。

      ⚠⚠ 事故的绝大多数来自这里

      Thariq 的观察：弱模型时代大家把精力全砸在 "known knowns" 的精细化上——例子塞满、"别做 X" 列个清单。Fable 时代最贵的两格挪到了下面两行：隐性和雷区。下面六件工具就是把这两格提前显影的杠杆。

      四抓 Unknowns 的六件工具：动手前撬开雷区#

      — PRE-IMPLEMENTATION PROBES

      前面提到，Fable 的一个物理事实是：它烧 token 快、走得深——如果它跑了几十步之后你才发现"这不是我想要的"，那几十步的成本是不可逆的。所以最划算的动作是在 coding 开始之前先做几个便宜的探针，把 unknown knowns 和 unknown unknowns 显出来。以下六件按使用顺序：

      01

      盲区扫描 Blindspot Pass

      让 Claude 先自主扫一遍"我可能没考虑到的边角"、"和现有代码的隐性冲突点"。它会给你一份 checklist，你只需要打勾/否决。

      治 unknown unknowns · 用 立刻，先于写 spec

      02

      发散设计 Brainstorm

      要求它给出 "3–5 个 wildly different design directions"。不是"更好的一版"，是心智模型完全不同的替代方案。逼它离开你的默认锚点。

      治 你被自己 default 锁死 · 用 定方向前 30 分钟

      03

      反向问询 Interview Me

      让它采访你——但要求"只问 impact 最高的问题"。类似 /grill-me，但强制排序。你会发现自己被问出很多你嘴里默认但从没落笔的东西。

      治 unknown knowns · 用 需求刚落地时

      04

      参考对齐 Use References

      尤其在迁移类任务里：给它一份"你希望产出长这样"的既有实现作为参照。迁移期最容易踩的是"两侧同名不同物"，参考实现能压住 60% 的漂移。

      治 从 A 迁到 B 时的隐性契约 · 用 spec 写完后

      05

      决策日志 implementation-notes.md

      让 agent 边做边写一个"我替你做的欠规约决定"的运行日志。事后 diff 比翻 3000 行代码快 10 倍——你只 review 决定，不 review 代码。

      治 agent 替你替下的默认决定 · 用 全程

      06

      反向测验 Quiz Me

      收尾前让它考你——但被测的是我的理解，不是它的。你答不上来的题就是"你以为你懂但其实没懂"的位置。

      治 你自己的"我以为我懂" · 用 merge/上线前

      合起来的姿态

      把这六个动作看成一条 pre-implementation 流水线：02 发散 → 01 盲区 → 03 反问 → 04 参考 → 05 边做边记 → 06 反测。前四步在 coding 开始前跑完，第五步全程跑，第六步在 merge 前跑。绝大多数返工出在跳过前三步就直接下场。

      五与「Unhobbling」是同一件事的两面#

      — TWO CLEAN-UP PASSES FOR A NEW MODEL CLASS

      keynote 第一段 Unhobbling Claude（解开脚镣）与第二段 Finding Your Unknowns（找出未知）常被人当成两个话题看，但 Thariq 本人在推里明说过——它们是同一次清理的两个方向。

      ←
      Unhobbling 清"过时的 known"

      砍掉旧模型时代堆上去的束缚。Thariq 一次砍了 Claude Code 系统 prompt 的 80%——大量"别做 X"和"照这个例子来"其实是在关闭想象力。给 context，不给 constraint；给背景，不给例子。

      ↑
      Finding Unknowns 显影"没意识到的未知"

      前面六件工具，全是"在 coding 开始前把不可见格显出来"。这一代模型的自主性大到需要它，弱模型时代就算不做也没事。

      ·
      共同前提 A new model class

      模型不是被 设计 出来的，是被 培育（grown）出来的——每一代能力谱都会重刷。所以每次遇到新一代模型，都要重跑一次这两个清理动作。这是 Anthropic 内部的口径。

      顺便提一下他给的一个反直觉证据：keynote 里那个"Pokemon 名字以 aw 结尾"的例子——普通 chat 模型答不上来，但 Claude Code 用工具能答上来。不是能力没有，是 harness 把能力掐住了。 这就是为什么 unhobbling 值得每一代做一次。

      六映射到我们的活：SeedAgent · 客服 · 视频质检#

      — WHAT THIS MEANS FOR US

      这套方法论最实用的地方，是它把我们已经在做的三条业务线里"最容易返工的那些位置"具体化了。逐条对齐一下：

      业务线 现在最容易踩的 unknown 直接可套的探针 落成 skill 的做法
      SeedAgent · RSI harness Harness 里长期沉淀的"隐性契约"（tool 调用顺序、错误重试策略、context 复用规则）——大部分没写进 spec。 Blindspot pass + implementation-notes.md 在 RSI loop 每一步之前跑一次 blindspot，事后归档 notes.md 作为 harness 演化的 changelog。
      电商客服 Agent（售前/售后） Unknown knowns 特别多："客户地址填错怎么办""退款超期怎么办""促销和会员权益冲突怎么算"——业务员脑子里都有，prompt 里全没。 Interview Me + Brainstorm 把 /interview-me 做成客服接线员的每日入岗仪式；每周让 agent 出 3–5 个 wildly different 应答风格，人工挑一次。
      添可社媒视频质检 · 达人 PE 三件套 Unknown unknowns 主要在"文化差异 + 平台差异"：一段在 TikTok 好使的钩子，在 IG Reels 或 Douyin 是雷点——但你在写 brief 时想不到。 Blindspot pass（跨市场版）+ Use references 让 agent 对每一份 brief 做一次"如果这条投放在 X 市场，会有哪些不适？"的 pass；references 库直接用历史爆款/翻车样本。
      家用机器人（情感陪伴 / 单臂） 物理侧的 unknown unknowns：光照、噪声、家庭成员的隐性称呼习惯——这些不会出现在任何 spec 里，但每一条都决定"是不是像个真机器人"。 Quiz Me + Brainstorm Quiz me 让工程师自己在 spec 提交前答 5 道场景题，答不出来的就是要补的 unknown。
      与 project 一句 lesson 对上号

      我们内部一直在讲 impact-per-token 而不是代码行数——这套方法论就是它的最佳操作化：前置澄清成本 vs 事后返工成本，Fable 越贵越 agentic，这个比值就越小。把预算前置到 unknowns 上，是当前阶段最高性价比的 harness 演进方向。

      七陷阱：什么会让这套失效#

      — WHERE IT BREAKS DOWN

      风险清单
      • 把 blindspot pass 当成"再问一遍需求"。 那样只能捞到 known unknowns。真正有价值的问法是"我可能没考虑到的约束和冲突是什么"——问它替你担心。
      • Interview me 走成填 20 题问卷。 Thariq 明确说"只问高 impact 的问题"——它需要被约束，否则会问出一堆"要不要支持导出 CSV"这种低价值题。
      • Brainstorm 出 3 个换汤不换药的方案。 要求 "wildly different" —— 心智模型不同（比如：一个走事件、一个走轮询、一个直接不做这个特性）。
      • implementation-notes.md 变流水账。 只该记"我替你替下的默认决定"（欠规约位置的选择），不该记"我改了 3 个文件"。
      • 把 unknowns 当成一次性动作。 每换一代模型都要重跑——上一代还是 unhobbling 主战场时，你写的很多约束在新一代变成了限速带。
      • 用弱模型做这套。 反问、盲区扫描依赖模型自身的"跨领域直觉"，弱模型只能给你重复你 prompt 的话；这一套的收益随模型强度指数上升，Fable/Claude 4+/Sonnet 5+ 才明显。

      八下一步行动（1–2 周）#

      — IMMEDIATE MOVES

      行动清单
      • 给 skill/insight-write/SKILL.md 加两个 subcommand：/blindspot-pass（开写前扫盲）和 /quiz-me（收尾前反测）。比再堆一层"写作规范"回报高得多。
      • SeedAgent harness 做一次 unhobbling 审计：把系统 prompt 里的"别做 X"清单和例子清单挑出来，评估哪些是"过时的 known"，砍一版 −50% 的对照测。
      • 客服 Agent 上一版 /interview-me：售前售后各选 2 位业务员，让 agent 反过来问他们，把答案沉淀成 unknown-knowns 补丁包。
      • 视频质检做一次 blindspot pass 试点：拿 5 条已经"上线后翻车"的历史素材，回测——如果 pass 能提前指出，就把它加进标准工作流。
      • 建立 implementation-notes.md 的机构惯例：每个 agent-driven 的 PR 都强制附一份。它是我们后续做审计和知识蒸馏的一等信号。

      九待验证 6 问#

      — OPEN QUESTIONS

      1. 盲区扫描的 recall 上限在哪？ 它能找出多少比例的 unknown unknowns？相同任务多次运行的稳定性如何？
      2. 六件工具之间的边际收益是否重叠？ 如果只做前两件，能覆盖多少？做全六件是不是过度？—— 需要一批带 ground truth 的历史事故复盘来量化。
      3. 换模型（Fable → Sonnet → GPT-5）时六件工具的相对权重会怎么漂？ Interview Me 依赖跨领域直觉，可能在闭源前沿模型间差异极大。
      4. 中文语境下 /interview-me 的问句质量。 Anthropic 训练语料里英文占比高，问题排序在中文上是否仍然按 impact 排？可能需要一层本地化 rerank。
      5. 与 spec-driven（Amazon Kiro 那种"规范驱动"）流派的关系。 Kiro 是"把 unknowns 全落进规范"，Thariq 是"用 agent 自己撬出 unknowns"——两者是替代还是互补？
      6. 把 implementation-notes.md 反哺到下一次任务作为参考的收益。 这是一种极便宜的组织级 RSI——但会不会在跨项目场景下反成噪音？
