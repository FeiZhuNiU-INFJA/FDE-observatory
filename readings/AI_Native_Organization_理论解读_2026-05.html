      一导读：为什么需要「理论库」#

      AI-native organization 不是单一作者的概念。McKinsey、BCG、Block、OpenAI、Deloitte 以及多家转型框架从不同切面描述同一场变革。若只读一家，容易：

      • 把 诊断（你在哪）当成 处方（该改什么）；
      • 把 架构（系统怎么建）当成 组织设计（编制怎么变）；
      • 把 激进 greenfield（Block）当成 legacy 企业 的唯一路径。

      本卷按 统一模板 解读 8 套主框架 + 2 套辅助模型，并在 §12–§13 给出关系地图与冲突说明。卷二在此基础上做融合判断与实现拆解。

      每章模板：来源与定位 → 核心概念 → 组织形态含义 → 与 AI-native 的关系 → 领导层要问 → 常见误区 → 适用边界。

      二McKinsey Agentic Organization · 五支柱#

      来源与定位#

      • 来源：The agentic organization（2025）；The state of organizations 2026（AI-first operating models）。
      • 解决什么问题：企业如何把 AI 从「个人提效工具」升级为 组织级运营范式——人类与 Agent 规模化协作，边际成本趋近于零。
      • 类型：处方类（五柱联动 redesign）+ 部分诊断（AI 悖论、double transformation）。

      核心概念#

      1. Agentic organization：工作流 AI-first；基本单元为 agentic team（2–5 人监督 50–100 Agent）。
      2. 五支柱：商业模式、运营模型、治理、人才与文化、技术与数据——同一 redesign 的五个切面。
      3. Work chart：按任务/outcome 交换，替代部分 org chart 的层级委派。
      4. Future-back + lighthouse：从 3–5 年目标体验倒推；先在一个高价值域五柱最小闭环。
      5. Double transformation：技术转型 + 组织转型必须同时进行。

      银行叙事（贯穿示例）：购房 journey 由房产/按揭/合规/合同/放款 Agent 链执行，人类 agentic team 督导；可延伸至装修、能源等 outcome 网络。

      组织形态含义#

      支柱从…到…
      商业模式Seat/工时Outcome、hyper-personalization、结果网络
      运营模型职能 siloAI-first 端到端 + agentic team + work chart
      治理周期审计实时嵌入、policy/critic agent、风险分级 gate
      人才文化ExecuteJudge：编排、评判、above the loop
      技术数据烟囱 ITRegistry、MCP/A2A、agentic mesh、semantic 数据

      与 AI-native 的关系#

      McKinsey 的 agentic organization 与 AI-native organization 在行业用法中高度重叠：均指运营模型重写，而非 AI overlay。本系列将 McKinsey 五支柱作为 legacy 大企业转型 最完整的处方框架。

      领导层要问#

      • 五柱是否 联动？还是只买了 Copilot（仅技术柱）？
      • 哪个 lighthouse 域可 AI-first 整段重设计？
      • Agent 自主边界与 outcome rubric 是否 explicit？

      常见误区#

      • 局部节点自动化 ≠ 生产力（BCG 与 McKinsey 共识）。
      • Agent 泛滥无 context = chaos。
      • 把 governance 等于 SOC2 认证。

      适用边界#

      • 适合：有复杂 legacy 系统、需渐进改造的大中型企业；需要董事会级「五柱 checklist」。
      • 不适合：作为唯一 greenfield 蓝图（见 Block）；不能替代具体技术架构设计（见 Frontier / Intelligence Architecture）。

      2.2 商业模式（Business Model）#

      定义：客户为 可验证 outcome 付费，而非 seat 或工时；Agent 使 marginal delivery cost 趋近于零，商业模式可从「卖时间」转向「卖结果网络」。

      维度LegacyAgentic
      定价单位Seat、FTE、工时Outcome、流程、订阅+结果
      价值主张标准化产品Hyper-personalization
      延伸单点交易Outcome 网络（购房→装修→能源）

      领导层要问：客户为「什么可验证结果」付费？Outcome rubric 是否 explicit？

      常见误区：只改定价话术、流程仍 seat 驱动；把 Copilot 席位当作新商业模式。

      2.3 运营模型（Operating Model）#

      定义：基本单元为 agentic team（2–5 人监督 50–100 Agent）；协作按 work chart（任务/outcome 交换）而非纯 org chart 层级委派。

      维度LegacyAgentic
      协调层级传话、会议Work chart + Agent 编排
      流程职能 silo 接力AI-first 端到端 redesign
      单元部门编制Agentic team + fleet

      银行叙事：购房 journey 由房产/按揭/合规/合同/放款 Agent 链执行，人类 team 督导；可延伸至装修、能源等 outcome 网络。

      领导层要问：哪个 lighthouse 域可整段 AI-first 重设计？Agent 与人类 gate 是否标注？

      2.4 治理（Governance）#

      定义：风险边界 嵌入 workflow，非事后补丁；三类 gate——自动（低风险）、规则（中风险，policy/critic agent）、人批（高风险：资金、权限、法律承诺）。

      维度LegacyAgentic
      节奏季度/年度审计实时、continuous
      对象人行为Agent action + human override
      机制合规 checklistGraduated autonomy + audit trail

      领导层要问：自主边界是否 written？100% 高风险 action 是否过 gate？

      常见误区：把 governance 等于 SOC2 认证；先上 Agent 后补治理。

      2.5 人才与文化（Workforce & Culture）#

      定义：角色从 Execute（执行）→ Judge（编排、评判、关系、above the loop）；JD、晋升、考核须反映 outcome ownership 与 spec/validation 能力。

      维度LegacyAgentic
      核心技能领域执行编排、评判、spec 写作
      位置In the loopAbove the loop
      考核产出量、busyOutcome、gate 质量、fleet leverage

      领导层要问：JD 是否已重写？Junior 是否学 spec/review 而非仅工具点击？

      2.6 技术与数据（Technology & Data）#

      定义：Double transformation——技术架构与组织 OM 同步改；Agent registry、MCP/A2A 互操作、semantic/Agent 可消费数据、agentic mesh。

      维度LegacyAgentic
      集成烟囱、点对点Registry + 标准连接器
      数据报表/仓库Agent 可 query 的 semantic layer
      部署项目制Platform + eval 闭环

      领导层要问：数据 Agent 能否消费？Registry 是否知道「谁在生产」？

      常见误区：只买 Copilot（仅技术柱）；Agent 泛滥无 shared context = chaos（McKinsey 警告）。

      2.7 五柱联动#

      flowchart TB

      2.8 Workshop 自检（1–5 分）#

      #问题1=否5=是
      1五柱是否在同一 lighthouse 域联动？

      失衡诊断：技术≥4、运营≤2 → T1.5；治理≤2 → 不敢规模化。

      2.9 与 T0–T3 对应#

      T 阶段五柱典型状态
      T1 / T1.5技术柱 spread（Copilot/单 Agent）；运营/治理/人才滞后
      T1.5→T2运营+五职能 explicit；治理 gate 嵌入 lighthouse

      三AI-Native Transformation Framework · T0–T3 成熟度#

      来源与定位#

      • 来源：AI-Native Organization · Reference Framework。
      • 解决什么问题：组织（及个人）处于 AI 转型的 哪个阶段；为何 T1.5 是多数企业陷阱。
      • 类型：诊断类。

      核心概念#

      阶段组织状态关键特征
      T0→T1AI 叠加个人用工具；流程不变
      T1.5AI 广、流程旧人饱和于 validation；质量漂移；多数企业 2026
      T1.5→T2工作流重设计五类结构职能 explicit；规格与验证门 deliberate
      T2→T3AI-native 默认新角色进编制；compression 完成

      个人 tier 与组织 level 映射：T1≈Level1 AI-Assisted；T2≈Level2 AI-Integrated；T3≈Level3 AI-Native。

      T3 个人工作单元：Context → Clarification → Execution → Validation → Recovery；cycle 从 sprint 级压到 feature 级（天）。

      组织形态含义#

      • T1.5：编制不变，但每人多了一层「审 AI 输出」——组织 名义上 AI 化，实质上 human 瓶颈。
      • T2：出现 Workflow Architect、Specification Owner 等 跨部门职能。
      • T3：部门墙模糊；价值集中在 spec 边界与 validation 边界。

      与 AI-native 的关系#

      T 阶段回答 「在哪」；McKinsey 五支柱回答 「改什么」。T1.5→T2 的门槛 = lighthouse 域 AI-first 重设计 + 五职能 explicit。

      领导层要问#

      • 我们自评 T 几？依据是什么（流程、编制、治理各一项证据）？
      • T1.5 持续多久？验证工作占一线时间多少比例？

      常见误区#

      • 把「全员 Copilot 席位」当作 T2。
      • 个人 T3、组织 T1——个人英雄无法拉动组织 level。

      适用边界#

      • 适合：转型启动诊断、对内对齐「为何投入大产出平」。
      • 不适合：替代五支柱的改造清单；Framework 偏方法论站点，案例少于 McKinsey/BCG。

      四五类结构职能（Direction / Specification / Validation / Agent Ops / Trust）#

      来源与定位#

      • 来源：AI-Native Transformation Framework · AI-Native Organization。
      • 解决什么问题：AI-native 组织 按什么切分工作——不是按部门，而是按「人不可替代的功能」。
      • 类型：处方类（岗位/职能设计）。

      核心概念#

      职能做什么典型角色
      Direction目标、意图、优先级领导、DRI
      Specification把 know-how 写成 Agent 可执行契约Specification Owner、Workflow Architect
      Validation风险分级 gate、质量判定Governance Specialist、Evaluation Engineer
      Agent Operations运行/监控 fleet、工具、上下文、再校准Agent Supervisor、DevOps/Data Engineer
      Trust & Human Relations客户/监管/员工关系、escalation、伦理Player-Coach、合规、CS 升级

      关键句：Legacy 按 Engineering/Sales/CS 分；AI-native 按 Function 切——同一角色可跨部门（Workflow Architect 最明显）。

      组织形态含义#

      • 编制从「职能头数」→ 五职能覆盖度（很多 legacy 组织 heavy Direction、weak Specification/Validation）。
      • T2 标志：五职能 deliberate；T3：五职能 first-class 编制。

      与 AI-native 的关系#

      五职能是 McKinsey「人才+运营+治理」的 可操作分解；与 Block 三角色正交（Block 讲问责结构，五职能讲工作类型）。

      领导层要问#

      • 五职能中哪几个 缺失或 implicit？
      • Specification 质量谁 owner？Validation 是事后还是嵌入？

      常见误区#

      • 只招 Agent Supervisor，无 Specification Owner → Agent 无 good spec。
      • Validation 全堆在一线（T1.5 验证瓶颈）。

      适用边界#

      • 适合：设计新 JD、agentic team charter、流程 RACI。
      • 不适合：替代 Block/McKinsey 的汇报与 team 结构决策。

      五Intelligence Architecture（智能架构）#

      来源与定位#

      • 来源：The AI-Native Architecture 等 practitioner 归纳。
      • 解决什么问题：若不用层级协调，用什么替代？——三层（或五层）系统架构。
      • 类型：架构类。

      核心概念#

      三大块（对外叙事）：

      1. 闭环运营模型：感知 → 综合 → 边界内决策 → 度量 → 更新规则（小时级）。
      2. 可查询知识层（Queryable Intelligence）：组织所知结构化、可检索、可版本化。
      3. 软件工厂：人写 spec/测试；Agent 执行迭代；主观/新颖处 hybrid。

      展开五层（实现视角）：

      • Governance layer：自主边界、HITL、日志、置信度阈值。
      • Tool layer：确定性能力（查库、发消息、改 ticket）。
      • Learning layer：对照 expectation 更新规则；政策变更须人批。

      一句 loop：Operations 产生 signal → Intelligence 变 knowledge → Software factory 变 product → 产生更多 signal。

      组织形态含义#

      • 中层协调职能 → 被 intelligence architecture 吸收（与 Block world model 同构）。
      • 人类移到：spec、judgment、relationship、policy approval（human layer）。

      与 AI-native 的关系#

      Intelligence Architecture 是 「系统长什么样」；McKinsey 五支柱是 「组织怎么改」。技术柱 + 运营柱落地时，常采用此架构 vocabulary。

      领导层要问#

      • 知识是否 queryable（非仅存在 SharePoint/PDF）？
      • Learning loop 周期是小时还是季度？
      • 软件工厂是否有 机器可验证 的验收标准？

      常见误区#

      • 只有 RAG，无 governance/learning → 静态知识堆。
      • 把 software factory 等于「全员 vibe coding」无验证门。

      适用边界#

      • 适合：CTO/架构师设计 target architecture；研发型 AI-native 组织。
      • 不适合：直接给 HR 做编制设计（需配合 Block/McKinsey）。

      六Block · 四层架构 + 三角色#

      来源与定位#

      • 来源：From Hierarchy to Intelligence（Block 官方，2026-03）；Amir Elion 解读。
      • 解决什么问题：能否用 AI 替代层级的协调功能（而非仅提效个人）？
      • 类型：架构类 + 组织设计（最激进的 public-company 实践）。

      核心概念#

      四层结构：

      层内容
      Capabilities原子能力（支付、发卡、借贷…）—— building blocks，无 UI
      Intelligence Layer按客户/时刻组合 capabilities，proactive
      InterfacesSquare、Cash App 等交付面
      World ModelsCompany world model + Customer world model（机器可读、持续更新）

      三角色（assignments，非汇报层级）：

      角色职责人类不可替代
      IC + Agent构建与运营判断、品味、创意
      DRI90 天 own 跨职能 outcome；可 pull 资源Ownership、问责
      Player-Coach在岗构建 + 培养人人际、教练

      Goose：开源 agentic harness（类似内部 Claude Code/Codex），Agent 运行时。

      核心反转：传统公司 intelligence 分布在人+层级；Block 模型 intelligence 在 system，人在 edge。

      组织形态含义#

      • 删除永久中层（Block 公开披露大幅压缩 middle management）。
      • World model 替代 manager 的 信息路由与对齐；DRI 替代 优先级与资源 pull；PC 替代 纯路由型 manager。
      • Remote-first + artifact 文化 是前提（决策/代码/设计/plan 机器可读）。

      与 AI-native 的关系#

      Block 是 Intelligence-Native 形态 B 的参照（见卷二）；与 McKinsey agentic team 兼容处：人都 above the loop；冲突处：Block 弱化 org chart，McKinsey 保留 work chart 网络。

      领导层要问#

      • 我们的工作 artifact 是否 machine-readable 到可建 world model？
      • 若删除协调层，DRI 授权是否真实？
      • 这是 cost optimization 还是 structural redesign？（Block 自问）

      常见误区#

      • 无 artifact 文化却抄三角色 → 对齐真空。
      • 把 Block 减员叙事当唯一 KPI。

      适用边界#

      • 适合：数字原生、remote-first、工程/产品 artifact 密集组织；局部域试验 world model。
      • 不适合：强监管、强物理现场、legacy ERP 为主的传统制造/政务（宜用形态 A）。

      七BCG · AI-First Company + Functions of the Future（FoF）#

      来源与定位#

      • 来源：Design Your Company for AI, Not AI for Your Company（2026）；What Corporate Functions of the Future Will Look Like；Agents Accelerate the Next Wave（含 MIT 合作研究）。
      • 解决什么问题：为何 95% AI 部署未产生 meaningful value；企业职能如何重组。
      • 类型：处方类。

      核心概念#

      1. AI-first organization：工作围绕 agent 动态协调系统 组织，而非 primarily human roles/hierarchies。
      2. Small cross-functional teams：人+Agent+数据+技术，端到端 own outcome（与 McKinsey agentic team 同构）。
      3. Autonomous execution vs automation：目标是对 outcome 的自主执行，非在旧流程上贴 automation。
      4. Shared AI platform：「Freedom within a frame」——统一基建，P&L 在框内创新。
      5. FoF 七 building blocks（摘要）：跨职能流程 redesign → 叠 AI/agents → 对齐数据系统 → 新角色（AI product owner、governance manager）→ 领导力适配 human+agent。
      6. Trust Protocol / Graduated Autonomy：分级自治；清晰 risk taxonomy、测试、监控、resilience。
      7. 数据点：45% AI leaders 预期减少 middle management；58% heavy adopters 预期三年 governance 根本变化。

      组织形态含义#

      • 职能未来 更 lean、更 automated、AI speed。
      • Work 不穿过部门链，而穿过 统一 workflow（planning→reporting，sourcing→payment）。
      • 新角色：AI product owner、model governance manager、talent data strategist 等。

      与 AI-native 的关系#

      BCG 与 McKinsey 高度互补：BCG 更强 FoF 序列、shared platform、95% 失败统计；McKinsey 更强 五柱 checklist 与银行 narrative。本系列将 BCG 作为 「为何必须 redesign 公司而非加 AI」 的论据库。

      领导层要问#

      • 是否有 shared platform，还是每部门自建？
      • 流程是否 跨职能 mapped 后才上 Agent？
      • Trust Protocol 中自治级别是否 defined？

      常见误区#

      • 个体/单职能 AI（BCG 指出的 main failure mode）。
      • 把 automation 当 transformation 对外宣称。

      适用边界#

      • 适合：集团职能转型（HR/Finance/Ops）；平台化 IT 治理讨论。
      • 不适合：小团队无需 FoF 全称；细节需结合 McKinsey 治理柱。

      八OpenAI Frontier · 企业语义操作系统#

      来源与定位#

      • 来源：Introducing OpenAI Frontier（2026-02-05）。
      • 解决什么问题：Agent 分散部署 → 集成碎片化；如何让 Agent 拥有 与人类员工同等的 business context。
      • 类型：架构类 + 平台实践（overlay 路线）。

      核心概念#

      1. Frontier = 企业 Agent 平台，非模型 API：build / deploy / manage AI coworkers。
      2. Shared business context（Universal Semantic Layer）：连接 data warehouse、CRM、ticketing、内部 app；Agent 共享 语义层。
      3. Agent 管理类比 HR：onboarding、identity、permissions、performance review、boundaries。
      4. Model-agnostic + 开放标准：可接 OpenAI/Anthropic/Google；不强制 replatform。
      5. Multi-agent orchestration + eval loop：生产级 Agent 需持续评估。
      6. Overlay 定位：坐在现有 SaaS 之上，非替换 ERP/CRM——与 Block greenfield 形成对照。

      组织形态含义#

      • 组织级 institutional memory 沉淀在 semantic layer（非个人 prompt）。
      • 「个人 prompt 资产」→ 组织流程资产（与 McKinsey 运营模型、BCG shared platform 同向）。
      • 技术团队 + 业务团队通过 同一 context 构建 Agent（democratization，受 governance 约束）。

      与 AI-native 的关系#

      Frontier 是 形态 A（Enterprise Agentic） 的技术 substrate 参照：legacy 企业不必推倒 ERP，但需 统一 context + agent registry。卷二 Layer3 的 Shared Context 主要由此 + Intelligence Architecture 融合而来。

      领导层要问#

      • Agent 是否 共享 context，还是每 Agent 一条集成？
      • Identity/permission 是否与 HR/IAM 对齐？
      • Eval 是否 上线门禁？

      常见误区#

      • 把 Frontier 当作「又一个 ChatGPT Enterprise」。
      • 无治理即大规模部署 coworkers。

      适用边界#

      • 适合：已有多云/多 SaaS、希望 overlay 的大型企业；平台选型讨论。
      • 不适合：无 digital footprint 的组织；替代 Block 式 world model 组织 redesign。

      九Deloitte · 技术组织重建#

      来源与定位#

      • 来源：AI and the future of the IT function（Tech Trends 2026）；Horizon Architecture Survey 2025 等。
      • 解决什么问题：AI 如何 重组 tech org 本身——不仅是业务侧 agentic。
      • 类型：使能类。

      核心概念#

      1. 78% tech leaders 预期五年内在 architecture workflow 广泛/深度集成 AI agents。
      2. Tech org 变 leaner、faster：AI 嵌入 architecture→delivery 各层。
      3. CIO 角色：从 IT 运维 → enterprise AI 架构师——提供标准化、安全、可扩展 building blocks。
      4. Cloud-native platform：API-first、self-service、platform engineering；云 = speed/flexibility 引擎。
      5. Project team → product/value-stream squads：与 agentic team 方向一致。
      6. Adaptive governance cycles：治理周期缩短，与 McKinsey 实时治理同向。

      组织形态含义#

      • Tech 组织率先 agentic 化：平台团队维护 Agent registry、连接器、安全基线；业务 agentic team 在平台上构建。
      • CIO 协调 「freedom within a frame」（与 BCG shared platform 同构）。

      与 AI-native 的关系#

      Deloitte 回答 「谁建底座、怎么建」——卷二 Phase 2「基建」的主要参照；不替代 McKinsey 业务侧五柱。

      领导层要问#

      • Tech 是 瓶颈还是 enabler？
      • 是否有 platform team 对 agent registry 负责？
      • Squads 是否 aligned to value stream 而非 project？

      常见误区#

      • 业务 Agent 满天飞，tech org 仍 project 制 + 烟囱。
      • 把 cloud 仅当 cost center，非 AI engine。

      适用边界#

      • 适合：CIO/CTO 规划；Enterprise Agentic 形态 A 的 IT 侧。
      • 不适合：纯业务读者（需先读 McKinsey/BCG）。

      十辅助模型 A：五层能力阶梯#

      来源与定位#

      • 来源：AI Journal 等归纳（The AI Transformation Reckoning）。
      • 类型：诊断类。

      五层#

      层级形态特征
      1Chatbot响应问题
      2Copilot单 workflow 内 assist
      3Agent跨工具完成 task
      4Multi-agent system专用 Agent 协作
      5AI-native operating model围绕 human-agent 协作 redesign work

      要点#

      • 第 5 层才触及 组织 OM；1–4 层可在 legacy OM 上运行。
      • 与 T1–T3 映射：T1≈1–2；T1.5≈3 扩散但 OM 未改；T2–T3≈4–5。

      十一辅助模型 B：Automation / Augmentation / Transformation#

      来源与定位#

      • 来源：AI Journal 同上。
      • 类型：诊断类（战略诚实性）。

      三选一（非阶梯）#

      战略做什么价值组织影响
      Automation用 AI 做已知任务更快省时间OM 不变
      Augmentation增强人类决策/创造提质OM 基本不变
      Transformation改 workflow / BM / 体验竞争优势改变组织是什么

      关键句：三者不是楼梯；选错战略名称会导致 预期与交付 mismatch（「自动化项目」被叫成「转型」）。

      领导层要问#

      • 董事会/OKR 对应哪一个？证据是什么？
      • 若选 Transformation，是否分配了 OM redesign 预算与 owner？
