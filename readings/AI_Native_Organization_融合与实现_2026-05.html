      一导读#

      卷一逐条解读 10 套框架。本卷做三件事：

      1. 融合：提炼各理论的「最大公约数」，形成对 AI-native 组织的统一理解。
      2. 预期：描述 2027–2028 两种典型形态及可观测指标。
      3. 实现：六阶段路径、每阶段交付物、失败模式对照。

      文中 【共识】 标注多源一致判断；【推断】 标注本卷融合推断，供读者自行验证。

      Part A：融合理解

      二A1. 三层统一模型#

      【共识】各框架可收敛为三层——战略与价值、运营与人、智能底座：

      flowchart TB subgraph L1 ["Layer 1 · Strategy & Value"] Maturity["T1–T3 与五层阶梯诊断"] Strategy["Automation / Augmentation / Transformation"] BM["Outcome 与商业模式"] end subgraph L2 ["Layer 2 · Operating & Human"] Team["Agentic Team"] Func["五类结构职能"] Roles["Block 三角色或编制问责"] end subgraph L3 ["Layer 3 · Intelligence Substrate"] Context["Shared Context 或 World Model"] Fleet["Agent Fleet 与编排"] Gov["Governance 与 Trust Protocol"] Learn["Learning Loop"] end L1 --> L2 L2 --> L3 L3 -->|"signal 与 knowledge"| L1
      层回答什么卷一主要来源
      Layer 1为何改、改到哪、卖什么T0–T3；Auto/Aug/Trans；McKinsey BM；BCG autonomous execution
      Layer 2谁负责、怎么协作McKinsey OM；五职能；Block 三角色；BCG cross-functional teams
      Layer 3靠什么跑、如何治理与学习Intelligence Architecture；Frontier；Deloitte platform；BCG Trust Protocol

      Layer 间约束：缺 Layer 3 → Agent 无 context（McKinsey chaos）；缺 Layer 2 规格/验证 → T1.5 验证瓶颈（Framework）；缺 Layer 1 战略诚实 → 自动化项目被叫转型（AI Journal）。

      三A2. 融合版定义（一句话）#

      【推断】AI-native organization 是一种运营模型：组织将 可验证的业务 outcome 作为价值与协作单位；由 小规模跨职能人类 在 规格、验证、信任 边界内，监督 共享上下文上的 Agent 舰队 执行端到端工作；治理与学习 嵌入流程而非事后补丁；人类默认处于 above the loop（意图、判断、关系、政策变更），Agent 处于 within the loop（执行、综合、监控、常规决策）。

      这与任一单理论的区别：

      • 比 McKinsey 更强调 Layer 3 shared context（来自 Frontier / Intelligence Architecture）。
      • 比 Block 更保留 legacy 系统与 work chart（适合多数企业）。
      • 比 Framework 更强调 商业模式与平台（BCG/McKinsey），不只个人 tier。

      四A3. 与 Legacy / T1.5 的本质差异（可观测）#

      【共识+推断】满足 ≥5 条 可认为走出 T1.5、进入 T2 通道：

      #可观测特征理论依据
      1至少 1 条流程 AI-first 端到端重设计（非仅加 Copilot）McKinsey、BCG 95%
      2存在 agentic team 或等价单元，KPI 绑 outcome 非职能McKinsey、BCG
      3Specification + Validation 有 explicit ownerFramework 五职能
      4Agent 动作 风险分级 + 审计可追溯McKinsey Gov、BCG Trust Protocol
      5Shared context 或 world model 覆盖 lighthouse 域主数据/流程Frontier、Block
      6新 Agent 上线绑 eval / regressionMcKinsey TD、Frontier
      7JD/晋升含 编排与评判，非仅工具使用McKinsey WF、Block

      T1.5 典型：仅满足 0–2 条（多为 Copilot 席位 + 个人 hero）。

      五A4. 两种形态预期（2027–2028）#

      【推断】未来不会单一形态。主流为 A 为主、B 为局部试验：

      形态 A：Enterprise Agentic（主流大企业）#

      维度预期
      描述McKinsey 五柱 + agentic team + Frontier 式 overlay + ERP/CRM 仍为真相源
      组织Work chart 网络；middle layer 压缩 30–45%（BCG/MIT 区间）
      技术Shared semantic layer；Agent registry；MCP/连接器；eval 闭环
      理论来源McKinsey、BCG、Frontier、Deloitte
      适合金融、制造、零售等 legacy 深、合规强

      形态 B：Intelligence-Native（激进 / 数字原生）#

      维度预期
      描述Block 式 company world model + 三角色 + intelligence 在 system
      组织永久中层极少；DRI 90 天制；artifact 机器可读
      技术Proactive intelligence layer；capabilities 组合；Goose 类 harness
      理论来源Block、Intelligence Architecture
      适合Remote-first、软件/金融科技、artifact 文化成熟

      混合策略【推断】#

      • 集团 = 形态 A 为 default。
      • 创新 lab / 新 BU = 形态 B 试验 world model。
      • 禁止：全集团无 context 狂上 Agent（McKinsey chaos）。

      Part B：形态预期 · 2028 目标态 Checklist

      按五维给出 T3 / 形态 A 成熟 lighthouse 的可验收状态（可在单域先达成）：

      B1. 编制#

      • Agentic team（或等价）成为 lighthouse 域 默认单元
      • 该域 middle coordination 角色减少或改责（非纯路由）
      • 五类职能中 Specification、Validation 有 named owner

      B2. 流程#

      • ≥1 条流程 AI-first 文档化（含 Agent 步与人类 gate）
      • 跨职能 handoff 通过 work chart / 统一 workflow，非邮件/层级
      • BCG 式：sourcing→payment 或同等 E2E 至少一条打通

      B3. 治理#

      • Graduated autonomy 表：低/中/高 action 对应 auto / rule / human
      • 100% 高风险 action 过 gate（资金、权限、对外法律承诺）
      • 单笔 Agent 决策可 replay（context + tool + outcome）

      B4. 技术#

      • Lighthouse 域有 shared context layer（Frontier 或自建 semantic layer）
      • Agent registry：谁在生产、版本、owner
      • Eval/regression 为 promote 门禁

      B5. 商业#

      • 至少 1 个 outcome 指标 与定价/内部 chargeback 试点挂钩
      • 客户或内部 SLA 用 可验证 rubric，非仅 seat

      Part C：如何实现 · 六阶段路径

      六C0. 总原则#

      【共识】

      • No partial optimization（McKinsey、BCG）
      • Future-back + lighthouse first（McKinsey）
      • Double transformation：组织与 tech 并行（McKinsey 2026）
      • 战略诚实：Automation/Augmentation/Transformation 三选一（AI Journal）

      七阶段 0：诊断（2–4 周）#

      目标：知道在哪、选哪种战略、选哪个 lighthouse。

      动作说明
      T0–T3 评估组织 + 抽样团队；证据：流程/编制/治理各 1 项
      五层阶梯定位多数业务在 1–3 层；目标域要到 4–5
      Auto/Aug/Trans 决策董事会/OKR 对齐真实战略
      Lighthouse 选型高价值、可度量、数据可达、领导层愿改流程

      交付物：

      • 成熟度评分表（T + 五层 + 五柱 1–5 分）
      • Lighthouse 选型 memo（1–2 页：域、outcome、系统、风险）
      • 战略声明（三选一）与资源承诺

      理论依据：Framework T；AI Journal；McKinsey lighthouse。

      八阶段 1：设计（4–8 周）#

      目标：AI-first 流程蓝图与职能/gate 设计。

      动作说明
      E2E 流程 mapping消除/自动化/简化/保留人工（BCG FoF 序列之先）
      AI-first redesignAgent 步 + human gate 标注
      五职能 RACI尤其 Specification、Validation owner
      Outcome rubric 草案与 Layer 1 商业模式对齐
      Agentic team charter2–5 人、outcome、与 fleet 关系

      交付物：

      • AI-first 流程图（As-Is / To-Be）
      • Gate 矩阵（action × 风险 × 处理方式）
      • Outcome rubric 与验收标准草案
      • Team charter 模板（含 DRI 若适用）

      理论依据：McKinsey OM；BCG process redesign；Framework 五职能。

      九阶段 2：基建（2–4 月，与阶段 3 并行启动）#

      目标：Agent 能跑、有 context、可观测。

      动作说明
      Shared context / semantic layer连接 lighthouse 域核心系统（Frontier 思路）
      Agent registry注册、版本、owner、权限 scope
      连接器 + identityMCP/API；最小权限
      Trace + cost与 business task 关联
      Platform squadDeloitte：CIO/platform 提供 building blocks

      交付物：

      • Context layer 设计 doc（数据源、刷新、权限）
      • Registry 最小实现 + 命名规范
      • Trace 字段标准（task_id、agent_id、outcome、tokens）

      理论依据：Frontier；Intelligence Architecture；Deloitte。

      十阶段 3：治理（与阶段 2 并行）#

      目标：敢规模化；符合 Trust Protocol。

      动作说明
      Graduated autonomy 表BCG Trust Protocol
      Policy / critic 机制嵌入 workflow
      Eval + regression 集promote 门禁
      审批/回滚政策与 Agent 配置变更
      Audit 要求满足监管/内控

      交付物：

      • Autonomy 分级表 + escalation 路径
      • Eval rubric + 首批 regression cases
      • 变更管理流程（P0/P1/P2 或等价）

      理论依据：McKinsey Gov；BCG Trust Protocol。

      十一阶段 4：编制（3–6 月）#

      目标：人与组织跟得上。

      动作说明
      JD 更新Execute→Judge；五职能写入
      ReskillJunior：spec/review；Senior：workflow architect
      Pilot agentic teamlighthouse 域运行 1 个 team
      考核口径outcome + 编排；非仅 busy
      可选 Block 试验创新域试 DRI 90 天 + 三角色

      交付物：

      • 关键岗位 JD 修订包
      • Pilot retrospective（outcome、gate 命中、人工占比）
      • 培训大纲（spec 写作、eval 解读、escalation）

      理论依据：McKinsey WF；Block；Framework T3 个人 loop。

      十二阶段 5：扩展（6–18 月）#

      目标：T2→T3；lighthouse 复制；商业试点。

      动作说明
      复制 playbook第 2、3 个 lighthouse 域
      FoF 扩展BCG：跨职能 unified workflow 增多
      Outcome 定价/chargeback 试点Layer 1 闭环
      可选形态 B labworld model 局部试验
      持续 learning loopIntelligence Architecture

      交付物：

      • 复制 playbook（流程+gate+registry 模板）
      • Outcome 商业实验设计（指标、争议、清算）
      • 年度 maturity 复评（T + 五柱）

      理论依据：McKinsey lighthouse；BCG FoF；Framework T3。

      十四C7. 失败模式 ↔ 理论对照#

      失败模式表现违反的理论/原则

      Agent 混沌fleet 无 steeringMcKinsey context warning；无 Frontier layer

      平台缺失每函数自建BCG shared platform

      十五C8. 给决策者的融合三句话#

      1. 【共识】 AI-native 不是 Copilot 普及，而是 Layer 1–3 联动 redesign；多数卡在 T1.5，因只改了 Layer 3 的工具、未改 Layer 2 流程与职能。
      2. 【推断】 2028 主流形态是 Enterprise Agentic（A）：agentic team + shared context + ERP 仍在；Block（B）是极限参照，适合 lab。
      3. 【共识+推断】 落地顺序：诊断 → 设计 → 基建∥治理 → 编制 → 扩展；lighthouse 单域五柱最小闭环，再复制。
